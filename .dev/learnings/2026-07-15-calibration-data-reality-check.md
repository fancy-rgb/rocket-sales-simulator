# 매출 시뮬레이터 — 정확도 개선용 데이터 실측 (1단계)

날짜: 2026-07-15
목적: 모델 개선(spec 착수) 전에 "앵커링·캘리브레이션에 쓸 실데이터가 실제로 얼마나 있는가"를 실측.
배경: challenger 검토에서 앵커링·캘리브레이션이 모두 과거 데이터 의존인데 표본 수가 미검증이라는 지적 → 이 문서로 해소.

## 1. 시뮬레이터 자체 DB (mlykalmpqalxhwrfoaow / `scenarios` 테이블)

| 구분 | 건수 |
|---|---|
| 전체 행 | 27 |
| 시뮬레이션(is_actual=false, 미삭제) | 23 |
| 실적(is_actual=true) | **1건 — 그것도 삭제됨(deleted=true)** |

→ **결론: 시뮬레이터가 원래 의도한 "실적 탭 입력 → 시뮬 vs 실적 비교" 캘리브레이션 루프는 사실상 사용되지 않고 있다.** 저장된 27건은 전부 사전 시뮬레이션(가정 입력)이고, 캘리브레이션 대조군이 될 실측 결과가 0건이다.

부수 관찰: 저장된 시뮬 시나리오들의 입력값 자체가 이미 넓게 흩어져 있다 — 참석률 40~55%, 결제CVR 3~7%로 사용자가 매번 다르게 가정하고 있다는 뜻(=현재 자유 입력이 벤치마크 부재로 감에 의존 중임을 방증).

## 2. 운영 콘솔 DB (biepgnnushywnwfdegxy) — 진짜 실적 소스

### 2.3 유료/오거닉 믹스 실측 (challenger 우려 1 검증)

```
ad_registrants/registrants 비율 (n=82)
평균 82.4% · 최소 21.5% · 최대 100.0%
```

→ **믹스가 실제로 21.5%~100%까지 크게 흔들린다.** challenger의 "blended CVR이 믹스 변화에 취약하다"는 우려가 가정이 아니라 **실측으로 확인된 사실**. 예: 로드옥션(2026-03-18)은 ad 100%인데 결제CVR 6.9%, 같은 로드옥션(2026-07-01)은 attend_rate가 34.2%로 다름 — 같은 파트너도 회차마다 크게 요동침.

### 2.4 참석률·결제CVR 실측 변동폭 (시뮬레이터 고정 기본값과 대조)

실제 웨비나 20건 샘플(2026-02~07):
- 참석률(attend_rate): **26.7% ~ 98.8%** — 시뮬레이터 고정 기본값 **40%**는 이 범위의 하단 근처
- 결제CVR: **1.6% ~ 15.9%** — 시뮬레이터 고정 기본값 **10%**는 중상단

→ 두 지표 모두 편차가 매우 커서, 단일 고정 기본값(40%/10%)으로 예측하면 실제 결과와 몇 배씩 어긋날 수 있음이 확인됨.

### 2.5 발견된 구조적 데이터 갭 — "FT참석(완주)" 단계는 실측 불가

시뮬레이터 모델은 참석→**FT참석(완주)**→결제 3단계인데, `webinars` 테이블에는 참석(attendees)·구매(buyers)만 있고 **완주자 수를 담는 컬럼이 없다.** 즉 이 중간 단계는 실측 데이터로 앵커링·검증할 방법이 현재 없음 — 계산에서 빼거나, 새 계측이 필요하거나, "참석→결제" 2단계로 단순화해야 함.

## 3. 종합 판정 (challenger 3대 미확정 전제 실측 결과)

| challenger 우려 | 실측 결과 |
|---|---|
| 카테고리별 벤치마크가 과적합 아닌가 | **부분적으로 맞음** — 상위 4개 업종만 유효(n≥13), 하위 3개는 표본 부족 → 전체 pooled 필요 |
| 유료/오거닉 분리가 실익 있는가 | **실익 있음, 실측 확인** — 믹스가 21.5~100%로 크게 흔들려 blended CVR 오염이 실제로 발생 |
| 캘리브레이션에 쓸 실적 데이터가 있는가 | **시뮬레이터 자체 DB엔 없음(0건). 다만 운영 콘솔 `webinars`에 114건 존재** — 단, 이 둘은 서로 연결이 안 되어 있어 시뮬레이터의 "실적 탭" 기능은 사실상 죽어있고, 진짜 실적은 다른 프로젝트 DB에 쌓이고 있음 |

## 4. 이번 실측이 2단계(모델 개선 spec)에 주는 함의

1. **"실적 탭 캘리브레이션 루프"는 설계를 다시 짜야 한다** — 원래 계획(시뮬레이터 자체 실적 탭 활용)은 데이터가 없어 무의미. 대신 **운영 콘솔 `webinars` 114건을 소스로 벤치마크 산출** 방향으로 전환 필요(프로젝트 간 연동 또는 수동 반영 방식 결정 필요).
2. **업종 벤치마크는 상위 4개만, 나머지는 pooled 기본값**으로 현실적 범위 설정.
3. **유료/오거닉 분리는 실익이 실측으로 확인**됐으므로 2단계 설계에 포함해야 함.
4. **FT참석(완주) 단계는 현재 실측 불가** — 모델에서 이 단계를 어떻게 다룰지(유지+미검증 명시 / 단순화) 2단계 설계에서 결정 필요.
5. 카테고리 어휘 불일치(CLAUDE.md 7개 코드 vs `webinars.industry` 실제 값) — 2단계에서 매핑 정리 필요.
