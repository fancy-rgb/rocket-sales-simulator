## 1. 배경·목적

### 1.1 문제 제기
매출 시뮬레이터는 결정론적 퍼널 곱셈(광고신청→총신청→참석→FT완주→결제)으로 거래액을 예측한다. 단위경제(VAT·PG수수료·수수료배분)는 정확하나, 핵심 전환율 3개(참석률 40%·FT참석률 80%·결제CVR 10%)가 일반 고정 기본값이라는 지적이 있었다.

### 1.2 challenger 독립 검토 결과 (확정 결함 2건)
1. **유료·오거닉을 합친 뒤 단일 전환율 적용** — 유료 트래픽은 전환이 낮은데, 광고비를 키우면(유료 비중↑) 모델은 같은 비율로 선형 확대해 거래액을 구조적으로 과대추정
2. **결정론적 점 추정** — 웨비나 1회 이벤트·참석자 N이 수십~수백 규모라 이항 변동성이 큼. 점 추정 하나로 보여주면 이 도구의 목적("무리한 기대 방지")과 정면 배치

### 1.3 데이터 실측 결과 (`.dev/learnings/2026-07-15-calibration-data-reality-check.md`)
- 시뮬레이터 자체 DB의 "실적 탭" 데이터: **0건 사용 가능** (유일한 1건도 삭제됨) — 원래 계획된 캘리브레이션 루프는 죽어있음
- 운영 콘솔(rocket-launch-dashboard) `webinars` 테이블: **114건**, 참석률 26.7~98.8%·결제CVR 1.6~15.9%로 실제 편차가 매우 큼
- 유료/오거닉 믹스: 21.5~100%로 크게 요동 → challenger 우려 1이 실측으로 확인됨
- 업종별 표본: 상위 4개(창업·부업24·뷰티23·부동산15·교육13)만 충분, 나머지(마케팅·디자인·취업, n=1~2)는 부족
- FT참석(완주) 단계: 실측 컬럼 없음. 단, `webinars.offer_reach`(오퍼도달자) 컬럼이 이미 스키마에 존재 — liveklass-bm의 웨비나 퍼널 지표 계산기(2026-07-13)와 연결되면 향후 이 단계의 실측 프록시가 될 예정 (현재 0건 채워짐)

### 1.4 성공 기준 (검증 가능)
1. 유료비중이 달라지면 결제CVR 제안값도 그 구간의 실측 평균으로 달라진다 (믹스 구간 pooled 방식으로 확인 — §5)
2. 결과 화면에서 결제자 수를 "표본추출 노이즈 + 웨비나 간 실측 변동"을 함께 반영한 범위로 확인할 수 있다 (토글)
3. 업종 선택 시 실데이터 기반 참석률·결제CVR이 자동 제안된다
4. 표본 부족 업종은 전체 평균으로 대체되고, 사용자에게 그 사실이 안내된다
5. 기존 저장 시나리오 23건이 변경 후에도 동일한 계산 결과로 재현된다 (§9 — 신규 입력 필드가 없어 자동으로 보장됨)

## 2. 작업 범위 경계

### 2.1 이번에 구현
- 시뮬레이터 자체 Supabase에 `benchmarks` 테이블 신설
- 운영 콘솔 `webinars` → `benchmarks` 집계 스크립트 (Claude 수동 실행, cron 아님)
- `calcSim()` 결제CVR 제안값을 **유료비중 구간(tertile) pooled 평균**으로 대체 (계산식 자체는 불변 — 제안값 소스만 변경)
- 결과 화면에 "표본추출 노이즈 + 실측 웨비나 간 변동" 결합 범위 토글 ("범위 보기")
- 업종 선택 UI + 자동 제안(표본 부족 시 전체 평균 fallback + 안내)
- FT참석 필드에 "실측 데이터 없음" 라벨 + `benchmarks`에 `ft_rate` 행 예약(현재 미사용, 미래 자동 활성화 대비)

### 2.2 이번에 제외 (별도 의제)
- 참석률·FT참석률의 유료/오거닉 분리 (결제CVR만 유료비중 구간 반영 — 참석률도 같은 이유로 세그먼트 직접 관측이 불가하나, 매출 왜곡 기여도가 결제CVR보다 작아 이번 범위에서는 제외. §5 참고)
- `webinars.offer_reach` 실제 채우기 파이프라인 구축 (liveklass-bm 계산기 → dashboard DB 연결은 별도 프로젝트/의제)
- 시뮬레이터 UX 시각 개선(2026-06-02 spec) — 이 spec과 독립적. 계산 로직 변경(이번 spec)과 표현 변경(기존 spec)은 서로 다른 트랙이나, 이번 계산 변경으로 결과 화면 마크업이 일부 바뀌므로 **이번 spec을 먼저 구현 후 UX 개선 적용 순서 권장**
- 운영 콘솔 통합 (기존 ADR대로 계속 보류)

### 2.3 절대 건드리지 않음 (불변 보증)
- 시나리오 저장·불러오기·삭제·필터 기능 자체
- Supabase 인증·접근 제어
- 단위경제 계산(VAT·PG수수료·수수료배분) — 이미 정확

## 3. 데이터 흐름 · 아키텍처

```
운영 콘솔 DB (biepgnnushywnwfdegxy.webinars, 114건)
        │
        │  ① 오프라인 집계 스크립트 (Python, 서비스 계정)
        │     - 업종별 참석률·결제CVR 평균·표준편차·n 계산
        │     - 유료비중 구간(tertile: <82% / 82~91% / >91%, 실측 분포 기반)별 결제CVR pooled 평균 계산
        │     - 표본 n<10 업종은 is_fallback=true + 전체 평균값으로 대체
        │     - Claude가 사용자 요청 시("벤치마크 갱신해줘") 수동 실행
        ▼
시뮬레이터 자체 DB (mlykalmpqalxhwrfoaow.benchmarks) — upsert
        │
        │  ② 클라이언트 JS가 같은 프로젝트 키로 조회 (기존 scenarios와 동일 방식)
        ▼
업종 드롭다운 선택 → 참석률·결제CVR 자동 제안 (유료비중 구간에 맞는 값, 직접 수정 가능)
```

**교차 프로젝트 키 노출 방지**: 시뮬레이터 클라이언트 코드는 운영 콘솔 DB에 직접 접속하지 않는다. 집계는 오프라인(서비스 계정)에서만 수행하고, 결과만 시뮬레이터 자체 DB로 옮긴다.
**서비스 계정 키 취급**: 집계 스크립트의 운영 콘솔 서비스 계정 키는 `.env.local`(또는 기존 마이그레이션 스크립트가 쓰는 `service-account-*.json` 패턴)로 로컬 보관, `.gitignore` 등록 확인 후 사용 (`git-and-security.md` §4).

**왜 세그먼트(유료/오거닉) 직접 분리 대신 구간(tertile) pooled 방식인가 (v1.1 수정 배경)**: `webinars` 테이블은 구매자 수(`buyers`)가 웨비나 전체 합계 1개뿐이라, 유료 신청자 중 몇 명이 샀는지 오거닉 신청자 중 몇 명이 샀는지 나눠 기록되어 있지 않다. 즉 "유료 CVR"·"오거닉 CVR"을 직접 관측할 방법이 없다(v1.0의 확정 결함). 대신 각 웨비나의 **유료비중**(광고신청÷총신청, 이미 기존 입력값으로 계산 가능)을 실측 분포(p33=82%, p66=91%)에 따라 3구간으로 나누고, **그 구간에 속한 웨비나들의 blended CVR을 그대로 pooled 평균**해 벤치마크로 쓴다. 회귀로 두 CVR을 역산하지 않으므로 통계적으로 더 투명하고, "유료비중이 높을수록 실제로 CVR이 낮아지는 패턴"을 그대로 반영한다.

## 4. `benchmarks` 테이블 설계

```sql
create table benchmarks (
  id uuid primary key default gen_random_uuid(),
  industry text not null,           -- 8개 업종 값 + '전체'
  mix_bucket text,                  -- 'low'(<82%) | 'mid'(82~91%) | 'high'(>91%) | null(attend_rate 등 믹스 무관 지표용)
  metric text not null,             -- 'attend_rate' | 'buy_cvr' | 'ft_rate'(예약, 현재 미사용)
  n integer not null,
  mean numeric not null,
  stddev numeric,                   -- 웨비나 간 실측 표준편차 (§6 범위 계산에 사용)
  is_fallback boolean default false,
  updated_at timestamptz default now()
);
```

- `attend_rate`는 `mix_bucket = null`(업종 단위로만 집계 — §2.2 결정), `buy_cvr`는 `mix_bucket` 3구간 각각 집계
- 구간 경계(82%/91%)는 82건 실측 데이터의 tertile — 라운드 넘버 임의 지정 아님

- RLS: 기존 `scenarios`와 동일 정책 재사용 (읽기는 인증 사용자, 쓰기는 스크립트 서비스 계정만)
- `ft_rate` 메트릭 행은 이번엔 스크립트가 생성하지 않음(소스 데이터 0건) — 향후 `webinars.offer_reach`가 채워지면 같은 스크립트에 로직만 추가해 자동 채워지는 구조로 설계(코드 변경 최소화)
- **null 업종(26건) 처리**: 업종별(`industry` IS NOT NULL) 집계에서는 제외하고, `industry='전체'`(pooled) 집계에는 포함한다 — 업종 미상이어도 결제CVR·참석률 자체는 유효한 관측치이므로.

## 5. 계산 로직 변경 (`calcSim`)

**계산식 자체는 불변** — challenger 지적으로 재확인: 세그먼트별 CVR을 실측할 수 없으므로, v1.0처럼 계산식을 "유료비중×CVR_유료 + 오거닉비중×CVR_오거닉"으로 바꾸지 않는다. 기존 그대로 유지:
```
결제자 = FT완주 × 결제CVR(단일값)     ← 변경 없음
```

**바뀌는 것은 "결제CVR의 제안값 소스"뿐**:
- 업종 선택 + 현재 입력된 광고신청/오거닉신청으로 **유료비중 계산** → 해당 tertile 구간의 pooled 결제CVR을 `s-cvr` 필드에 제안값으로 채움 (기존 단일 필드 그대로, 새 입력 필드 추가 없음)
- 사용자가 광고비·CAC·오거닉 신청자 값을 바꿔 유료비중이 다른 구간으로 이동하면, 결제CVR 제안값도 그 구간 값으로 갱신(입력은 여전히 자유 수정 가능 — 강제 아님)
- 참석률·FT참석률은 기존과 동일하게 업종 단위(믹스 무관) 공통값 사용

## 6. 범위(밴드) 표시

**v1.0 대비 수정 배경**: 이항 표본추출 노이즈만 반영하면 실제 범위보다 훨씬 좁게 나온다 — 실측 데이터에서 참석률 stddev 17.3%p·결제CVR stddev 10.8%p로 **웨비나 간 실제 변동**이 표본추출 노이즈보다 훨씬 크다(과소추정은 이 도구 목적과 정반대). 두 변동원을 함께 반영한다.

- 기본 노출: 기존과 동일하게 점 추정만 표시
- "범위 보기" 토글 클릭 시: 표본추출 노이즈(이항분포) + 웨비나 간 실측 변동(stddev)을 합산해 범위 계산
  ```
  p = 참석률 × FT참석률 × 결제CVR   (현재 입력/제안값 기준 유효 전환율)
  상대분산 = (참석률_stddev/참석률)² + (결제CVR_stddev/결제CVR)²   (독립 가정, 오차 전파)
  σ_between = p × sqrt(상대분산)         ← 웨비나 간 실측 변동
  σ_binomial = sqrt(총신청 × p × (1-p)) ÷ 총신청   ← 표본추출 노이즈 (비율 기준)
  σ_p = sqrt(σ_between² + σ_binomial²)   ← 두 변동원 합산(quadrature)
  결제자_범위 = 총신청 × p ± 1.96 × 총신청 × σ_p   (95% 구간, 0 이상 clamp)
  ```
- 참석률_stddev·결제CVR_stddev는 `benchmarks.stddev` 값(업종/구간 미상이면 전체 pooled stddev: 17.3%p / 10.8%p) 사용
- "예상 결제자 6.4명 (95% 범위: 2~11명)" 형태로 노출 — v1.0 계산보다 범위가 넓어짐(의도된 결과). 거래액도 같은 배율로 범위 환산

## 7. 업종 벤치마크 UI

- 시뮬 탭에 "업종" 드롭다운 신설 (8개 값 + 전체, 기본값 = **미선택**) — CLAUDE.md 기존 7개 코드(MKT/BIZ/FIN 등)와 `webinars.industry` 실제 값(창업·부업/뷰티·라이프스타일 등) 간 매핑표 작성 필요. **CLAUDE.md도 이번 spec 구현 시 함께 갱신** (실제 채택 값 기준으로 카테고리 분류 체계 절 수정 — 문서 불일치 방지)
- 선택 시 `benchmarks`에서 해당 업종의 `attend_rate` + (현재 유료비중에 맞는 구간의) `buy_cvr` 조회해 `s-attend`·`s-cvr` 필드에 제안값 자동 채움 (사용자가 직접 수정 가능, 강제 아님)
- `is_fallback=true`인 경우 필드 옆에 "⚠️ 표본 부족 — 전체 평균 참고값" 안내 표시
- **업종 미선택(기본 상태)**: 자동 제안을 적용하지 않고 기존 하드코딩 기본값(참석률 40%·결제CVR 10%)을 그대로 유지 — 신규 사용자·빈 시나리오의 초기 화면이 이번 변경으로 달라지지 않도록 보장

## 8. FT참석(완주) 단계 처리

- 필드·계산 그대로 유지
- UI 라벨에 "🟡 실측 데이터 없음 — 경험치 입력값" 안내 추가 (다른 두 전환율과 신뢰도가 다름을 투명하게 표시)
- `benchmarks` 테이블에 `metric='ft_rate'` 행 스키마는 준비하되 이번엔 데이터 없음 — 향후 `webinars.offer_reach` 채워지면 집계 스크립트에 로직만 추가해 활성화 (모델 구조 변경 불필요)

## 9. 하위 호환

v1.1에서는 **계산식·입력 필드가 전혀 바뀌지 않는다** — 결제CVR 필드(`s-cvr`)는 여전히 단일값이고, 계산식도 기존 그대로다. 바뀌는 것은 이 필드에 채워지는 "제안값의 출처"뿐(업종/유료비중 기반 자동 제안 추가)이므로, 기존 저장 시나리오 23건은 **마이그레이션 없이 그대로 로드·재현된다.** (v1.0에서 계획했던 `cvrPaid`/`cvrOrganic` 필드 분리·하위호환 로직은 §5 재설계로 불필요해짐.)

## 10. 검증 기준

- **제안값 로직 검증**: 유료비중이 다른 입력 3개(예: 유료비중 낮음/중간/높음)로 업종 선택 시 `s-cvr` 제안값이 올바른 구간값으로 바뀌는지 확인. 계산식 자체(`결제자 = FT완주 × 결제CVR`)는 불변이므로 별도 회귀 테스트 불필요
- **하위 호환 검증**: 기존 23건 시나리오 재로드 시 거래액·결제자 등 결과값이 변경 전과 완전히 동일한지 확인 (입력 필드·계산식 불변이므로 자동 보장되어야 함)
- **범위(밴드) 계산 검증**: §6 수식(오차 전파 + 이항 노이즈 합산)을 수동 계산해 코드 출력과 대조. v1.0 이항 단독 계산보다 넓게 나오는지 확인(의도된 변화)
- **업종 자동 제안 검증**: 표본 충분 업종(예: 뷰티)과 부족 업종(예: 마케팅) 각각 선택 시 올바른 값·안내 문구 확인
- **집계 스크립트 검증**: `benchmarks` upsert 후 건수·값이 운영 콘솔 원본 집계(§1.3·tertile 82%/91%)와 일치하는지 대조
- **서비스 계정 키 검증**: 스크립트 실행 전 키 파일이 `.gitignore`에 등록되어 커밋 대상에서 제외되는지 확인
