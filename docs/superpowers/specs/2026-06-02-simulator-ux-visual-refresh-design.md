## 1. 개요·목적

매출 시뮬레이터(`index.html`, 단일 HTML)의 **표현/시각 품질**을 개선한다.
입력 방식·계산 로직·데이터 저장·기존 기능은 **그대로 둔다** (운영 중 서비스, 매니저 사용성 보호).

### 핵심 목적 재정의 (이번 설계의 기준)
이 도구는 **파트너사와의 조율·분석 자료**다 (설득·홍보 자료가 아니다). 파트너에게 화면 캡처로 공유되며, 세 가지 용도를 가진다.
1. **현실적 규모 합의 / 무리한 기대 방지** — "이 광고비 → 각 퍼널 단계 전환값 적용 → 이 거래액"의 인과를 함께 보며 현실적 거래액 규모에 눈높이를 맞춘다. 과대 기대를 막는 게 핵심.
2. **사전 목표 설정** — 합의된 시뮬레이션이 곧 사전 목표 지표(거래액·전환율 등)가 된다.
3. **회고 시 달성도 분석** — 웨비나 종료 후 실적을 입력해 **시뮬(목표) vs 실적**을 비교, 각 지표가 얼마나 달성됐는지 본다.

→ 시각 톤은 **차분하고 신뢰감 있는 분석 톤**. 축하·설득식 과장 스타일을 지양한다. 거래액은 크게 보여주되 "현재 입력 기준 예상"으로 담담하게.

### 성공 기준 (검증 가능)
1. 퍼널·결과·비교 3개 영역이 운영 콘솔 "Soft Gradient Glass" 디자인 언어로 통일된다 (분석 톤).
2. 퍼널에서 "이탈" 부정 강조가 사라지고, 결과 화면의 핵심 지표가 **거래액(현실 예상)**으로 읽힌다.
3. 시나리오 비교표에서 빨강=최저 부정 프레임이 사라진다.
4. **시뮬 vs 실적 비교(회고 달성도)** 뷰가 같은 디자인 언어로 정리되어, 목표 대비 달성도가 한눈에 읽힌다.
5. 입력 폼·계산·저장·불러오기·삭제·필터 동작이 **변경 전과 동일**하다 (로컬 캡처 회귀 비교로 확인).

## 2. 작업 범위 경계

### 2.1 이번에 구현 (표현·시각만)
- 퍼널 예측 렌더 (`funnelHTML`) — 시뮬·실적 탭 공통
- 매출 예측 결과 렌더 (`calcSim` 내 `#result-sim` 마크업) — 거래액 hero화
- 시나리오 비교 (`renderScenarios`) — 카드·표 스타일 정리
- **시뮬 vs 실적 비교 차트 (`hbarChart`, 회고 달성도 뷰)** — 같은 토큰으로 정리
- 전역 디자인 토큰(`:root` CSS 변수) 추가 + 기존 스타일 토큰 정합

### 2.2 spec에 저장만 — 구현 보류 (§6)
- 보수 / 현실 / 이상 **3안 구조** + 현실안 기본 + "추천(승자)" 폐기 + (반)자동 생성
- 운영 콘솔(`/tools/sales-simulator`) 통합 (→ `.dev/adr/2026-06-02_ux-first-integration-deferred.md`)

### 2.3 절대 건드리지 않음 (불변 보증)
- 입력 폼 구조·필드·라벨·`oninput` 핸들러
- 계산 로직 (`calcSim`/`calcActual` 수식)
- Supabase 저장/로드, 인증 게이트
- 기능: 시나리오 저장·불러오기(`loadScenario`)·삭제·필터(`setLaunchFilter` 등)
- DOM 컨테이너 ID (`#funnel-sim`·`#funnel-actual`·`#result-sim`·`#scenario-list`·`#scenario-compare-table`·필터 영역 ID) — 외부 참조 보존

> 변경은 위 컨테이너의 **innerHTML 마크업/CSS**에 한정. JS 계산·데이터 흐름·핸들러는 손대지 않는다.

## 3. 디자인 언어 (Soft Gradient Glass)

운영 콘솔 `rocket-launch-dashboard/src/styles/tokens.css`(2026-06-02 리스킨)를 단일 HTML에 이식한다.

| 역할 | 값 |
|------|----|
| 캔버스 배경 | `linear-gradient(135deg,#F6F5FA,#F1EFF6 55%,#F4F1F7)` |
| 글래스 카드 | `rgba(255,255,255,0.72)` + `backdrop-filter:blur(12px)` + border `rgba(255,255,255,0.9)` + radius 20~22px |
| Primary 바이올렛 | `#6E56CF` (hover `#5E46C4`) |
| 그라디언트 강조 | `radial-gradient(120% 120% at 0% 0%,#8B5CF6,#6D28D9 48%,#4F46E5)` |
| 큰 숫자 그라디언트 텍스트 | violet `#7C3AED→#5B21B6` / green `#059669→#047857` |
| 상태색 | success `#1FA463` / info(실적) `#2F6BFF` / danger `#E5484D`(최소 사용) |
| 그림자 | `0 10px 34px rgba(80,60,160,0.10)` |

> 시각 SoT = 본 spec + 승인된 목업 파일 (`.superpowers/brainstorm/74639-1780390363/content/`):
> `funnel-polished.html`(A 가로막대) · `outcome-funnel.html`(결과) · `scenario-compare.html`(비교).
> DESIGN.md 게이트는 운영 콘솔 토큰 + 승인 목업으로 충족 (design-decisions.md §0).

## 4. 화면별 설계

### 4.1 퍼널 예측 (`funnelHTML`, 시뮬·실적 공통)
**현재**: 단계가 가로 한 줄의 작은 숫자 + 파란 전환율. 줄어듦이 안 보임.
**개선**: **가로 막대 퍼널**.
- 각 단계 = 막대 1줄. 막대 길이 = 인원수 ÷ 최댓값(총신청) × 100%.
- 막대 안: 인원 숫자(흰색 굵게) + "명". 좌측에 단계명.
- 단계 사이: 전환율을 **담백한 보라 텍스트**(예: "참석률 55%")로만. **이탈 인원/빨강 강조 제거**.
- 마지막 "결제" 막대 = **초록 그라디언트**(성과 달성색), 나머지 = 바이올렛 그라디언트.
- 단계: 광고신청 → 총신청 → 참석 → FT참석 → 결제 (현행 순서 유지).
- 입력: `steps` 배열(label·n·rate) 그대로. 마크업/CSS만 교체.

### 4.2 매출 예측 결과 (`#result-sim`, 시뮬 탭)
**현재**: 거래액·공헌이익·예상정산 KPI 3개 동급 + 매출흐름 워터폴 + 수익배분 + 결제자/ROAS.
**개선**: **거래액을 핵심 지표로 — 단 분석 톤(과장 금지)**.
- 최상단 **거래액 강조 영역**: 라벨 "예상 거래액 (현재 입력 기준)" + 큰 숫자(예 `5,346만원`) + 서브("객단가 × 결제자 · 광고비 N만 투입 기준"). 축하식 화려한 배너가 아니라 **차분한 분석 카드** 톤 — 그라디언트는 절제해 사용.
- 그 아래 KPI 카드 3개: 예상 정산금액(파트너 수령액)·공헌이익·ROAS.
- 매출 흐름 워터폴·수익 배분·손익분기: **유지**, 토큰 색·간격만 정합. 손익분기는 "흑자/분기점 대비 배수"를 사실 기반으로(과장 없이).
- 계산값(gmv·revenue·contribution·expectedSettlement·roas·beGmv 등) 그대로 사용.

> 실적 탭(`calcActual`/`#result-...`)의 결과 영역도 동일 토큰으로 정합하되, 구조는 현행 유지(시뮬과 일관). 우선순위는 시뮬 탭.

### 4.3 시나리오 비교 (`renderScenarios`)
**현재**: 필터 3줄 + 카드들 + 비교표(초록=최고/빨강=최저). 카드·표 숫자 중복, 빨강 부정 프레임.
**개선 (카드·표 둘 다 유지, 스타일만 — 옵션 1)**:
- **필터**: 3줄 → **1줄 압축**(런칭·타입·작성자 칩 한 줄). 활성 칩 = 바이올렛 fill. 동작(`setLaunchFilter` 등) 불변.
- **카드**: 글래스 카드로 재스타일. 배지·이름·메타·스탯·**불러오기 버튼 위치 유지**. 실적 카드 delta 표기 유지.
- **비교표**: 빨강=최저 **삭제**. **최고값만** 담백 강조(보라 + ★). 각 셀에 **인셀 막대**(해당 지표 최댓값 대비 길이)로 크기 비교. 실적 열 = 파란 음영. 항목 그룹("규모/효율") 헤더.
- 정렬·필터·best 계산은 기존 로직 재사용(빨강 클래스만 제거, 막대 width 계산 추가).

### 4.4 시뮬 vs 실적 비교 — 회고 달성도 뷰 (`hbarChart`)
**용도**: 핵심 목적 ③ — 사전 목표(시뮬) 대비 실제(실적) 달성도 분석. 특정 런칭 선택 시 표시되는 가로 막대 쌍(시뮬/실적) 차트.
**현재**: 시뮬·실적 막대 쌍 + 음수 처리. 라벨·색이 토큰과 불일치.
**개선**:
- 시뮬 = 바이올렛, 실적 = 블루(`#2F6BFF`)로 토큰 정합. 막대쌍 비교 유지.
- **달성도 관점 보강(시각만)**: 실적/시뮬 비율을 담백하게 병기(예 "목표 대비 88%"). 과소달성도 빨강 경보가 아닌 **중립 회색/블루** 톤으로(분석 자료 성격).
- 계산·데이터(`sim`/`act` 객체, `hbarChart` 입력)는 그대로. 마크업/CSS만 교체.

## 5. 구현 방식 (단일 HTML)
- `<style>` 상단에 `:root` 디자인 토큰 변수 블록 추가.
- `funnelHTML()` 함수 본문(반환 마크업) 교체 + 관련 `.funnel*` CSS 교체.
- `calcSim()` 내 `#result-sim` innerHTML 템플릿에 hero 배너 추가 + KPI 카드 토큰화.
- `renderScenarios()` 내 카드/표 생성 템플릿 + `.compare-table`·필터 CSS 교체. 빨강(worst) 클래스 제거, 인셀 막대 추가.
- JS 계산·핸들러·데이터 접근 코드는 수정하지 않는다.

## 6. 보류 항목 (저장만, 이번 구현 제외)

### 6.1 보수 / 현실 / 이상 3안 구조
- 비교를 "경쟁"이 아닌 "합의" 도구로 — 보수 → **현실(기본 강조)** → 이상 스펙트럼.
- "추천(승자)" 배너 폐기, **현실안 기본 노출**.
- 생성 방식 미정(수동 3안 / 반자동 ± / 티어 태그) — 추후 결정.
- 보류 사유: 입력 방식 변화를 수반 → 운영 중 매니저 사용성 리스크. 별도 의제로 분리.

- `/tools/sales-simulator` 라우트로 통합 — P0(테스트 환경 분리) 이후. ADR 참조.
